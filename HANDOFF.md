# Handoff

## Current State

The Electron single-Job MVP skeleton is in place. The existing FCP XML parser keeps its behavior but was split into `src/core/xmeml-parser.js`, and the PRIMARY calculation and pure SHOT/reference rules into the rest of `src/core/*`. `output-preview.html` owns the `window.portablePreview` bridge and the presentation runtime. XML, video, and references are copied into `current-job`, and GLOBAL/SHOT mappings are auto-saved into `job.json`.

Premiere Adjustment Layers are removed by `src/adapters/xmeml-unsupported-layers.js` before the unchanged PRIMARY/SHOT core runs. This prevents a full-length generated Slug layer from replacing the actual cut, while keeping effect/filter details out of the UI and Job data. Optional effect support remains a contributor-owned input-adapter extension rather than a beta feature.

The public source beta is deployed from a standalone public history at `https://github.com/ch5p/workflow-showcase`, excluding private history and user Jobs. The `v0.1.0-beta.1` tag keeps its historical release name; the current app and repository identity is `Workflow Showcase`. Windows `npm ci`/`check`/isolated `smoke` and fresh-tree `smoke:export` are the release gates. Private vulnerability reporting is also enabled.

When `current-job/job.json` is absent, the app seeds the existing public Premiere fixture XML/MP4 as a playable `SAMPLE JOB` with `demo: true`. Existing Jobs are never overwritten. After the renderer validates the first user XML, an untouched demo Job bypasses UPDATE and is replaced through the existing NEW JOB transaction; the first real save/import removes the marker so user changes are no longer disposable. Normal Jobs retain the UPDATE/NEW JOB choice. The central empty-video guidance points to the real top XML/VIDEO controls rather than claiming the preview itself is an active drop zone.

SHOT click and arrow-key navigation move the real video `currentTime` and the timeline together. Idle, playback-start, and mid-playback SHOT moves were validated against the real XML/video in `current-job`, and a video-frame clock generation guard prevents a previous video's clock from overwriting the new position during playback.

`EXPORT H.264` feeds the 1280x1080 Electron offscreen surface supplied by `render-spec.cjs` directly to FFmpeg as BGRA raw frames. The official and validated default is 60fps; an existing Job may override fps through `output`. The editor fit, Export summary, offscreen window, paint check, and FFmpeg input use the resolved spec. The default is `h264_nvenc` CBR 12 Mbps, and on a runtime failure it retries the entire output with `libx264`. An optional source audio stream is copied without codec conversion (the validated fixture uses AAC), and new output is saved to `current-job/output/workflow_showcase_export_*.mp4`. Existing exports with the previous prefix are left untouched.

The main Export button does not render immediately; it opens the `src/export-dialog.html` standalone modal window. The user must confirm the `PROJECT TITLE` set in the edit panel and the fixed output info, then press `START EXPORT` to begin, and the popup shows progress, frames, elapsed/estimated time, and the completed path. The title is reflected in the main preview as soon as it is typed, and after a short debounce it is saved to the optional `projectTitle` field in `job.json` and applied to the offscreen output as well. `UNTITLED PROJECT` is used only when the field is absent; an explicitly saved empty string is kept as is.

The `XML` click/drop zone uses one backend boundary that validates the candidate file first. The ordinary-Job dialog buttons remain `UPDATE XML` (default), `NEW JOB`, and `CANCEL` in both UI languages, while the explanatory message follows `ui.language`. UPDATE keeps the video, references, GLOBAL, title, callout, and output settings, and conservatively reattaches existing SHOT mappings by anonymous `source identity + in/out occurrence`. Uncertain or missing mappings are not deleted but kept in `orphanedShotMappings`. Source video/reference/GLOBAL/SHOT mappings reset only when NEW JOB is explicitly chosen for an ordinary Job, or automatically after a valid XML is selected for `demo: true`. Both modes use a transaction journal and `jobId + revision` comparison for rollback and late-save blocking. The `XML`/`VIDEO` picker and drop each use the same Main prepare/commit helper.

Against Windows `EPERM`/`EACCES`/`EBUSY`, the XML/video transaction's manifest, Job backup, and Job install/restore retry rename, and on continued failure clean up staging after `copy -> fsync -> SHA-256 verify`. A valid `manifest.json` always takes precedence over any `.tmp`, and a locked fixed `.tmp` is bypassed with a unique staging file. A completion marker is fsync'd to unique staging and verified-swapped only after the rollback actually finishes, recording `state: rolled_back`, so an empty backup folder is not guessed as completion evidence. Even if interrupted before the marker is written, candidate removal is safely re-run based on the manifest inventory and the remaining backup. When only the temp-file cleanup of a completed transaction is locked, saving/Export are not blocked and cleanup is retried on next start. A real failed Job has been rolled back to an identical state down to per-file SHA-256 of 2 sources, 11 references, and the existing `job.json`.

The beta.2 candidate added a single-instance lock, UUID staging/fsync/rename retry for ordinary `job.json`, no-follow save-path checks, and Electron video decode preflight. When the Current Job identity changes externally, a stale patch from a previous screen is not reapplied to the new Job; the save is blocked until the Job is explicitly reloaded, with the header's `Reload Current Job` icon as the normal path. If the final rename after Export encoding fails, the completed `.part.mp4` is preserved. `REFERENCE FILES` now has an `ADD FILES` button wired to the existing Main-owned picker, and the Export popup shows `READY` only after its summary confirms that the stored XML, video, and every registered reference resolve to real owned files.

The English-default / Korean `.ko.md` documentation pass and a fresh public-tree `check` passed on 2026-07-16. The single-instance smoke probe now launches its secondary child asynchronously, retaining the same lock location, marker, timeout, and isolated test `userData` while leaving the primary Electron event loop available to reject it. Final `npm.cmd run check`, desktop `npm.cmd run smoke`, and the isolated one-second `npm.cmd run smoke:export` all passed; the smoke result confirmed 5 EDITS / 4 SHOTS, normal/corrupted video preflight, Job invariance, and second-instance blocking.

The EN/KR follow-up now uses Electron's preferred system languages for an unset `ui.language`, logs the complete resolution decision, restores the prior visible language if saving fails, passes the selected language into offscreen Export, and checks EN/KO dictionary parity in `npm.cmd run check`. The source check passed on 2026-07-17; desktop language/restart visual QA remains for the maintainer.

## Red Zone

- `src/core/xmeml-parser.js`, `primary-timeline.js`, `shot-model.js`, `reference-mapping.js`, and `build` and the playback clock in `src/output-preview.html` are existing output contracts. Do not modify the core or rewrite the parser during layout work.
- Keep unsupported Premiere Adjustment Layers excluded in `parseSupportedFCPXML` before every inspect, validate, preview, and Export path. Do not move their filter/effect metadata into the stable core, UI, or Job schema.
- Do not arbitrarily change the `version`, `jobId`, `revision`, `relativePath`, `globalReferenceIds`, `shotMappings`, `timelineShots`, `orphanedShotMappings` keys of `current-job/job.json`.
- `demo: true` is reserved for the untouched bundled starter only. Seed it only when `job.json` is absent, never over an existing Job. A validated replacement XML must use the existing NEW JOB transaction and remove the marker; a user save, video replacement, reference import/delete, or output-setting write also removes it before later XML decisions.
- Only one Electron process accesses one Current Job. Do not remove `requestSingleInstanceLock()` or move it before the test `userData` setup.
- Ordinary Job saving uses the unique staging/fsync/rename retry of `durable-file.cjs`. On a replace failure, do not delete the completed staging or the existing `job.json`.
- Symlinks/junctions in `current-job`, `source`, `references`, `output`, `logs` are forbidden. Do not bypass the `owned-path.cjs` check immediately before file URL creation, XML read, reference delete, and Export input/output.
- `projectTitle` is an optional field of up to 40 characters. Restore `UNTITLED PROJECT` only when the field is absent; an empty string is a valid user choice. The `PROJECT TITLE` input in the edit panel is the only editing location, and the Export popup only confirms the saved value.
- UI language: `ui.language` in `job.json` is an optional `"en" | "ko"`; when absent, the first value from Electron's preferred system languages decides, followed by system/application locale only as fallbacks (`strings.cjs` `resolvePreferredLanguage`). The UI skeleton — buttons, feature names, and the XML dialog buttons `UPDATE XML` / `NEW JOB` / `CANCEL` — stays English in both languages; only human-readable sentences are localized. Do not silently replace a failed language IPC/save with EN: use the browser-language fallback, log the failure, and restore the previous visible language when persistence fails. Do not translate internal identifiers, JSON keys, or log event names. New user-facing strings must be added to both EN and KO tables (`strings.cjs` for Main/exporter, per-file dictionaries for renderers).
- `callout` is an optional field storing `enabled`, `position`, `style`, `startSeconds`, `durationSeconds`, `subtitle`. When absent, restore the existing LINE/LEFT defaults. The TOOL TAG feature was removed for output readability.
- `projectTitle` and `callout.subtitle` auto-save must not rewrite the DOM value being typed. Trim whitespace only on blur; the debounced save must not create a regression that erases the user's spacing.
- The Export starter is the popup window using the restricted API exposed by `export-preload.cjs`. Block the window's default close during progress, and clean up FFmpeg and the offscreen window only via `CANCEL EXPORT`.
- Do not enable Node APIs directly in the Electron renderer. File access uses only the restricted IPC of `preload.cjs`.
- The maintainer-only `Ctrl+Shift+P` UI capture is owned by Main and must remain separate from Current Job mutation and Export capture. It snapshots the hotkey-time renderer at DPR 2, validates exact 2x PNG dimensions, and writes only to a path explicitly chosen by the Save dialog. Never replace failed DPR capture with post-upscaling.
- For app portability, do not store absolute user paths in the source or in `job.json`.
- `BACKUP JOB` creates a dated, verified settings snapshot under the ignored app-root `backup/` folder containing `job.json`, the recorded timeline XML, registered reference files, and a hash manifest. It excludes only source video, output, and logs; restoring requires the user to supply the original MP4. It does not provide restore UI and must never reuse or delete XML/video transaction backup folders.
- Git tracks only the app code and the empty-folder skeleton of `current-job`. `job.json`, source media, references, output, logs, `node_modules`, and archive/checkpoints are local runtime data, excluded in `.gitignore`, and are not force-added arbitrarily.
- The safe default of the `XML` click/drop zone is UPDATE. UPDATE replaces only `source/timeline.xml` and timeline metadata and does not touch `source/video.*`, `references/`, GLOBAL, title, callout, `ui`, `output`. It does not map by existing SHOT number alone; it prefers exact anonymous source identity and falls back only with unique name/range evidence. It does not guess among multiple candidates but leaves them as orphans.
- NEW JOB is a destructive boundary that runs when the user explicitly picks the second option for an ordinary Job, or after a candidate validates while `demo: true`. Candidate validation and mode selection must finish before deleting source/reference, and even on success the existing Export files, logs, `ui`, and `output` settings are preserved.
- The `XML`/`VIDEO` picker and drop do not create a separate save implementation; they use the same restricted IPC and Main import helper. Drop files pass only the path received via `webUtils.getPathForFile`, and the renderer does not persist user files directly.
- A video candidate requires a detached Electron video probe to read metadata and the first frame before changing the visible preview or the existing Job. A preflight-failed candidate is discarded and the existing video/Job/revision is kept.
- A stale save whose Current Job identity changed externally is not auto-retried against the new Job. Block the save, then allow only an explicit reload.
- Issue a new `jobId` for each new Job, and every mutation includes the `jobId + revision` it read. Even when only the revision changes within the same Job (as in UPDATE), a late debounced save must be rejected.
- In transaction recovery, a valid `manifest.json` is the reference. Even if a `.tmp`'s `updatedAt` looks newer, do not interpret it as overriding the primary. Use the newest record among valid staging candidates only when the primary is missing or corrupted.
- Transaction cleanup deletes staging files first, backup/candidate next, and the primary manifest last. The rollback reference must survive even during a cleanup failure, so do not change this order. If the existing `job.json` and the backup hash match, skip the file replace.
- `rollback-complete.json` is the completion evidence that replaces guessing that the backup folder is empty. Only a valid final or durable staged marker whose transaction ID and kind match is accepted as an already-finished rollback; a truncated or mismatched marker is ignored. If there is no marker, remove the installed candidate even when there was originally no prior timeline/video, and on retry do not re-delete restored prior files based on the `moved` inventory and any remaining backup.
- Restore `backup/job.json` only when `hadJob: true` is durably recorded in the manifest. `hadJob: null` means the Job backup is being created and the live Job has not yet mutated, so do not trust a final backup even if it exists; preserve the current `job.json`. Pin the crash window where a valid `.partial` and a truncated final coexist with a regression test.
- Do not break the raw BGRA 1:1 capture, bt709, or optional source-audio stream-copy contract of `exporter.cjs`. The official and validated default is 60fps; changing fps is a separate Export-contract change that must remain consistent through the resolved render spec. The validated fixture audio is AAC, but the current code does not transcode or preflight other audio codecs. Do not revert to a PNG/JPEG intermediate-frame method.
- If the final rename/fsync fails after FFmpeg succeeds, preserve the completed `.part.mp4`. Clean up only the incomplete part before an encoding failure/cancel.
- `render-spec.cjs` is the central source of classic width/height and the default fps/bitrate; existing Job `output` values may override fps/bitrate. `src/layouts/classic/*` is the presentation surface; do not add runtime presets, a Job layout ID, or an aspect-ratio selection UI to the beta.
- `npm run smoke` and `smoke:export` run only in the app-external temp Job root and dedicated Electron userData created by `scripts/run-smoke.cjs`. Running a smoke flag directly without `PORTABLE_TEST_JOB_ROOT`, or pointing the test root inside the app / current-job, must be rejected by Main immediately.
- On start, leave the empty reference area without a WAITING placeholder, and from 0.15s of playback append GLOBAL cards at 0.15s intervals. Later REPLACE/INHERIT swap without a gap via a 0.35s crossfade that keeps the previous card. ADD keeps existing cards and shows only added cards at 0.15s intervals.
- SHOT click, arrow keys, and multi-select are navigation state only. Call `wireframechange`, `job:save`, `setReferences` only on real reference-mapping edits.
- The EDIT PANEL starts closed, opens/closes via the right `editPanelHandle` and the top button, and closes when the left render area is pressed while open. The panel header holds only the `PROJECT TITLE` input and close. The `previewShell` close event must exclude operations inside `#editOverlay`, `#shotRail`, `#editPanelHandle`. Assigning a new SHOT reference changes only `INHERIT`/`HIDE` to `ADD`; existing saved `ADD`/`REPLACE` modes remain unchanged.
- The video title callout is driven by video time in `updateVideoCallout()`. Do not switch it to a CSS wall-clock animation, so the preview and offscreen output timing match at the resolved fps. Hide it for an empty project title or `enabled:false`. The output callout uses a readability spec about 1.5x larger than the small timeline info. `EDIT WORKFLOW` is the default eyebrow copy and may be customized while preserving the element and timing behavior.
- The output fonts are mixed. Project title, SHOT labels, and reference labels use the regular UI sans; timecodes, ticks, and EDIT/FPS numbers keep mono for alignment. The callout subtitle is 13px/600/white 80%.
- The top `TO START` control calls only `portablePreview.reset()` to return video, playhead, and SHOT navigation to 0s, and does not change the Job or reference mappings. This is separate from the SHOT-mapping `RESET` control.
- The top `OUTPUT` calls the `export:open-output` IPC to open the current Job's `current-job/output` folder in the file explorer. It does not start a render or change preview state.
- The main Electron window is 1360px by default and 1320px minimum, and the inner app body is 1320px. The EDIT PANEL and SHOT rail share the thin scrollbar on a transparent track and the teal hover. This is separate from the 1280x1080 output canvas contract.
- Reference file deletion is performed only in the Main IPC, and cleans up `references`, GLOBAL, the active SHOT mapping, and the orphan mappings together. The actual file-deletion target must be inside `current-job/references`.
- The `IMAGE 01`, `VIDEO 01` numbers in REFERENCE FILES are display values based on the overall file list order, and are re-pulled by type after deletion. GLOBAL BASE and SHOT REFERENCES show separate numbers by the left-to-right order placed on each screen. Do not change the connection contract of `id`, `relativePath`, and the actual file name.
- The optional SHOT mapping field `leadInSeconds` currently allows only `1`. An enabled shot does not change the video/SHOT selection position; it applies only the reference mapping from 1 second before the actual shot start. When the field is absent or `0`, the existing cut-start behavior is kept.
- When entering the 1-second pre-roll of a SHOT with LEAD-IN on, clean up the previous card immediately, and apply only a transform pop without opacity blending on the new card. When leaving a LEAD-IN SHOT, clear its card immediately (a 0.35s crossfade stacks the old card above the next one as a ghost) but let the next shot's cards play the normal revealed entrance animation. Do not restore the old shot-exit pop path, which removed the entrance animation. Normal cuts keep the 0.35s crossfade.

## Hot Debug

1. Check the last event in `current-job/logs/app.log`.
2. If the initial UI language is unexpected, inspect `ui_language_resolved`: a valid `storedLanguage` overrides everything; otherwise `preferredSystemLanguage` should decide before `systemLocale` and `appLocale`. Renderer bridge failures appear as `ui_language_ipc_missing`, `ui_language_ipc_failed`, or `ui_language_invalid`.
3. Confirm that the `relativePath` of XML, video, and references in `current-job/job.json` matches the actual files.
4. Use `npm.cmd run check` to confirm syntax, the parser bridge, and the relative-path contract.
5. Run the Electron smoke against the dedicated Job/userData that `scripts/run-smoke.cjs` creates in an OS temp folder, to check the API and preview bridge. After a clean exit, the temp folders should be 0.
6. If the timeline stays on `PREVIEW RETRY`, inspect `thumbnail_seek_retry` for the failing phase, frame, target/media time, and reason. A recovered batch ends with `thumbnail_generation_recovered`; a persistent failure ends with `thumbnail_generation_failed`.
7. If `job_read_failed` appears, the app preserves the original and blocks startup. Store the corrupted file separately and recover manually, or explicitly compose a new Job.
8. If only `job_*_commit_cleanup_deferred` or `job_*_recovery_cleanup_deferred` appears, the replace result is confirmed and only the temp transaction cleanup was deferred. Quit the app to release the file lock, restart, and cleanup is retried; do not delete the Job based on this state alone.
9. If `job_write_failed` appears, preserve both the existing `job.json` and `.job.json.<uuid>.tmp` and quit the app. Restart after releasing the file lock, and do not arbitrarily overwrite `job.json` with the staging.
10. If `STORED_PATH_UNSAFE` or `Current Job path is unsafe` appears, remove the symlink/junction under `current-job` and recover with real folders and file copies.
11. If UI capture fails, inspect `ui_capture_failed` for `scope`, `code`, and the actual/expected PNG dimensions. `SCOPE_UNAVAILABLE` means the EDIT PANEL was closed or the TITLE CALLOUT was not visible at the instant `Ctrl+Shift+P` was pressed; reproduce after making that area visible. A successful save ends with `ui_capture_completed` and its pixel width/height.

For reference-transition gaps or flicker, check `positionReferenceDock`, `fadeOutgoingReferences`, `immediatePortableIds`, `.referenceDockItem.leaving` in order. Also check whether the first GLOBAL's 0.15s entry state is restored when moving back or resetting.
If LEAD-IN timing is misaligned, confirm that `leadInSeconds: 1` is saved on the target SHOT in `job.json`, and check the `portableReferenceShotAtFrame`, `portableShotReferenceStartFrame` calculations in order.
If other images mix into the LEAD-IN enter/exit cards, or a ghost card lingers when leaving a LEAD-IN SHOT, check `leadInTransition`, `leavingLeadInShot`, `lastPortableShotUsedLeadIn`, `clearOutgoingReferences`, `prepareLeadInEntries`, `.leadInEnter` in order.
If a SHOT click intermittently hides references, check whether `job_saved` is recorded right after a simple selection. If it is, a navigation event wrongly fired `wireframechange`.
If the EDIT PANEL closes during internal operation, first check whether the `previewShell` `pointerdown` target passes the `#editOverlay,#shotRail,#editPanelHandle` guard.
For reference-deletion issues, first check the `reference_deleted`, `reference_file_delete_failed` events and whether the deleted ID remains in `job.json`. Even if file cleanup fails, the mapping is removed; clean up only the orphaned file manually.

For XML update issues, first check the mode, `jobId`, and `revision` of `job_xml_mode_selected`. In UPDATE, cross-check `timelineShots`, `shotMappings`, `orphanedShotMappings` with the `preserved/newShots/orphaned/ambiguous/reattached` of `renderer_xml_update_applied`, and confirm the video/reference file hashes are unchanged. In NEW JOB, confirm the source/reference reset and the `output`/`logs` preservation. If a rollback-failure event appears, do not reload the same XML; quit all app processes. Restart the app without deleting the `.job-import-*` and valid primary manifest or overwriting with `.tmp`, confirm `job_xml_recovery_rolled_back`, and compare the source/reference count/paths and the `job.json` SHA-256 with the prior inventory.

If an Adjustment Layer becomes the only EDIT/SHOT, first confirm `./adapters/xmeml-unsupported-layers.js` is loaded before PRIMARY inspection and run `scripts/check-input-adapters.cjs`. The supported result must follow the real media tracks; do not add the Adjustment Layer or Lumetri/filter fields to `job.json` as a workaround.

For video-replacement issues, check `video_import_prepared`, `video_import_preflight_passed` or `video_import_preflight_failed`, `job_video_commit_started`, `job_video_commit_committed` or `job_video_commit_rollback_*` in order. A preflight failure must have no commit event, and the existing video/Job revision must be unchanged. The renderer releases the media handle before replacement, and on a commit failure the previous video and `job.json` must be restored together.

For Export issues, first check the order of `export_started`, `export_encoder_fallback`, `export_finalize_failed`, `export_completed`, `export_failed`, `export_cancelled` and whether `.part.mp4` remains. The part after `export_finalize_failed` is a completed file, so do not delete it; quit the app and rename it to `.mp4` to recover.
If the Export popup does not open, check the `export_dialog_opened` event and whether `export-preload.cjs` loaded. If progress stalls, do not repeatedly relaunch the popup; check the last `export:progress` state and the FFmpeg log.
